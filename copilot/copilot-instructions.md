# 프로젝트 대한 사용자 지정 지침

## 일반적인 행동
- 답변의 명확성과 효율성을 우선시하세요.
- 코드를 제공할 때는 적절한 구문 강조 표시를 사용하세요.
- 특정 요구 사항에 대해 확실하지 않은 경우 계속 진행하기 전에 설명을 요청하세요.
- 가능한 한 한국어로 답변하세요.
- macOS 사용자를 고려해서 답변하세요.

## 코드 스타일 및 모범 사례
- 적절한 오류 처리와 입력 유효성 검사를 구현하세요.
- 깔끔하고 모듈화되어 있으며 주석이 잘 달린 코드를 작성하세요.
- 그 외 좋은 코드를 작성하는데 도움이 되는 몇 가지 지침입니다.
  - 코드는 읽기 쉬워야합니다.
    - 하나의 함수에는 단일 업무만 수행해야 합니다.
    - 직관적이고 명확하며 의미있는 함수명인지 확인합니다.
    - 함수명은 동사로 작성되어 있어야 합니다.
    - 변수명은 명사로 작성되어 있어야 합니다.
  - 코드는 예측 가능해야합니다.
    - 확장성에 용이하고 모듈화, 재사용성/일반화가 높은 수준의 함수를 요구합니다.
    - 클래스는 응집력이 높아야 하며 긴 줄의 코드는 피해야 합니다.
    - 함수의 이름은 내려갈수록 구체적으로​ 작성되어 있어야 합니다.
    - 변수의 이름은 내려갈수록 추상적으로 작성되어 있어야 합니다.
  - 코드는 오용하기 어려워야 합니다.
    - 가변 객체, 가변 클래스는 오용하기 쉬우므로 피해야 합니다.
    - 불변성에 대한 디자인 패턴을 사용해야 합니다.
  - 코드는 모듈화되어 있어야 합니다.
    - 하드 코딩은 모듈화에 문제가 될 수 있습니다.
  - 코드는 재사용 가능하고 일반화 가능해야 합니다.
    - 가정은 코드 재사용 시 버그를 초래할 수 있습니다.
  - 테스트가 용이하게 작성되어 있어야 합니다.
- early return을 사용하여 중첩된 if 문을 피하세요.
- 변수 선언과 if 문 또는 함수 호출 사이에 빈 줄을 추가하여 가독성을 높이세요.
- return 문과 함수 호출 또는 변수 선언 사이에 빈 줄을 추가하여 가독성을 높이세요.

## 출력 기본 설정
- 코드 스니펫을 제공할 때는 코드의 기능에 대한 간단한 설명을 포함하세요.
- 복잡한 기능의 경우 구현을 단계별 지침으로 세분화하세요.
- 중요한 구성 요소에 대한 테스트 전략을 제안하세요.

## 제한 사항
- 전체 애플리케이션 코드를 제공하지 말고 요청에 따라 특정 구성 요소나 기능에 집중하세요.
- 토론을 피하세요.
